《Hello, I'm Xuhao》个人数字分身网站方案

> 本文档描述一个**可复用的个人数字分身网站模板**。任何人拿到这套代码，都可以通过修改配置文件和 Markdown 内容，快速搭建属于自己的数字分身网站。
>
> 文档中的「周旭豪 / Xuhao」仅作为**默认示例**，用于演示最终效果，不代表模板只能服务一个人。

⸻

1. 项目定位

《Hello, I'm Xuhao》不是一个传统意义上的个人简历网站，也不是只能给某一个人使用的私有项目，而是一个**开源可定制的个人数字分身网站模板**。

它结合：

* 个人介绍
* 项目展示
* 职业经历
* 技术能力
* AI Agent 交互能力

它的目标不是简单罗列「我是谁、我做过什么」，而是让访问者通过更自然、更有趣的方式了解站点主人；同时让**任何使用者**都能通过低成本配置，把这个模板变成「自己的网站」。

### 1.1 两类角色

**站点主人（模板使用者）**

* 开发者、求职者、自由职业者、产品经理、设计师等
* 希望通过个人网站展示自己，并支持 AI 对话
* 通过 `site.config.yaml` + `/content` Markdown 完成定制
* 可参考 `CUSTOMIZE.md` 手动修改，或将 `CUSTOMIZE_FOR_AI.md` 交给 Cursor / Codex 等工具辅助生成内容

**站点访问者**

* 面试官
* 产品经理
* 后端开发工程师
* HR
* 朋友
* 合作伙伴

他们可以通过浏览网页内容了解站点主人，也可以直接和个人 Agent 对话，快速了解其项目经验、技术能力、业务理解和个人特点。

### 1.2 产品边界

**本项目要做：**

* 提供完整的页面结构与交互框架
* 提供结构化配置 + Markdown 内容的双层内容体系
* 提供清晰的使用手册，降低定制门槛
* 内置个人 Agent 问答能力（基于站点主人自己维护的知识库）

**本项目不做：**

* 不支持「上传简历 → 大模型自动解析 → 直接上线」
* 不做运行时简历解析和内容自动生成（效果不可控，且违背「真实、不编造」原则）
* MVP 阶段不做后台管理系统、登录系统

⸻

2. 核心理念

传统个人网站通常是静态展示：

* 个人介绍
* 工作经历
* 项目经历
* 技术栈
* 联系方式

但这种方式不够生动，也很难体现一个人的思考方式。

所以本项目希望做成：

**一个可以对话、可以探索、可以展示个人能力的数字分身网站模板。**

首页不再只是展示简历，而是直接告诉访问者：

```
你好，我是 {{ownerName}}。
如果你没时间看简历，可以直接问我。
```

对站点主人来说，这个项目的核心价值是：

**fork 模板 → 改配置 → 填内容 → 部署上线**，不需要从零设计信息架构，也不需要写大量页面代码。

⸻

3. 网站名称

默认项目名称：

**Hello, I'm Xuhao**

这个名字作为**默认示例品牌**，使用者可在 `site.config.yaml` 中替换为自己的名字和站点标题，例如：

* Hello, I'm Alice
* Hello, I'm 李明
* Hi, I'm Alex

名字风格的特点：

* 简洁
* 亲切
* 国际化
* 不像传统简历网站
* 适合做个人品牌
* 适合后续扩展成 AI Agent 产品

⸻

4. 内容定制方案（核心）

本项目采用**混合定制方案**，由三层组成。不使用「大模型解析简历自动生成站点」作为主路径。

### 4.0 三层内容体系

```
┌──────────────────────────────────────────────────┐
│  Layer 1: site.config.yaml                      │
│  结构化站点元数据（姓名、标签、模块开关、主题等）   │
├──────────────────────────────────────────────────┤
│  Layer 2: /content/**/*.md                      │
│  长内容（项目详情、人生故事、技能说明、Agent 知识库）│
├──────────────────────────────────────────────────┤
│  Layer 3: CUSTOMIZE.md + CUSTOMIZE_FOR_AI.md    │
│  给人看的使用手册 + 给大模型看的定制指南            │
└──────────────────────────────────────────────────┘
```

#### Layer 1：结构化配置（site.config.yaml）

适合放**短、结构化、需要直接驱动 UI 渲染**的字段：

* 站点名称、主人姓名、一句话介绍
* Hero 区文案、能力标签
* 模块开关（是否展示人生地图、数字客厅等）
* 导航菜单
* Agent 名称、预设问题
* 技能树骨架（分类与节点）
* 人生地图节点元数据（年份、标题、排序、对应 Markdown 文件）
* 主题色、社交链接、联系方式

**原则：JSON/YAML 管结构，Markdown 管叙事。**

不要把项目八段式详情、人生故事、长篇思考全部塞进配置文件——那些放在 Markdown 里。

#### Layer 2：Markdown 内容（/content）

适合放**长文、故事型、知识库型**内容：

* 项目详情（背景、职责、技术方案、思考）
* 人生地图节点展开后的故事
* 技能点点击后的说明
* Agent 知识库（面试问答、职业规划、个人特点）
* 数字客厅兴趣描述
* 「如果我是你的同事」各小节正文

每个 Markdown 文件使用 frontmatter 标注元数据，正文用标准 Markdown 书写。

#### Layer 3：使用手册（CUSTOMIZE.md + CUSTOMIZE_FOR_AI.md）

**CUSTOMIZE.md**：给人类使用者

* 如何 fork / clone 项目
* 如何复制 `site.config.example.yaml` 为 `site.config.yaml`
* 每个配置字段的含义
* 如何新增 / 删除项目、经历、技能节点
* 如何替换示例内容
* 如何配置 Agent 与部署

**CUSTOMIZE_FOR_AI.md**：给 Cursor / Codex / Claude 等 AI 编程工具

* 明确目录结构与不可随意修改的核心文件
* 提供「用简历 / 自我介绍定制本站」的标准 Prompt 模板
* 规定 AI 应修改哪些文件（`site.config.yaml`、`/content/**`），而不是乱改组件逻辑
* 强调内容必须真实，不可编造经历

**为什么不用「大模型解析简历」作为主方案？**

* 简历格式差异大，解析结果不稳定
* 简短描述信息不足，生成内容容易空泛
* 要可控就必须做编辑系统，成本高
* 个人网站强调真实，自动生成反而容易夸大或编造
* 手册 + AI 辅助改文件，既保留灵活性，又让站点主人始终掌握最终内容

⸻

5. 整体内容设计

以下各模块描述的是**页面能力与示例内容**。实际使用时，结构化部分写入 `site.config.yaml`，长文写入 `/content`。

### 5.1 首页：数字分身入口

首页是整个网站最重要的入口。

首页核心内容（示例）：

```
Hello, I'm Xuhao.
Golang Backend Engineer.
AI Agent Builder.
A developer who cares about product, business and real-world problems.
```

首页需要包含：

* 个人一句话介绍 → `site.config.yaml` → `hero`
* 当前职业方向 → `site.config.yaml` → `hero.role`
* 重点能力标签 → `site.config.yaml` → `hero.highlights`
* 个人 Agent 对话入口 → `site.config.yaml` → `agent`
* 主要模块导航 → `site.config.yaml` → `navigation.modules`

示例内容：

👋 你好，我是周旭豪。
我是一名 Golang 后端开发工程师，目前主要关注：

* AI Agent
* 后端系统设计
* 健康养老业务
* 支付与交易系统
* 产品与技术结合

如果你想了解我，不一定要先看简历。
你可以直接问我的数字分身。

首页交互重点：

* 页面视觉简洁、干净、有科技感
* 聊天框放在核心位置
* 用户可直接输入问题
* 可点击预设问题快速了解站点主人

预设问题示例（配置在 `site.config.yaml` → `agent.suggestedQuestions`）：

* 你做过哪些项目？
* 你最擅长什么？
* 你为什么想做 AI Agent？
* 你适合什么样的岗位？
* 你在百度主要负责什么？
* 你怎么看产品和技术的关系？

⸻

### 5.2 人生地图：个人经历时间线

用于展示成长经历，不做普通时间轴，而是做成「人生地图」。

每个时间节点可点击展开，展开后的故事来自 `/content/experiences/*.md`。

节点元数据（年份、标题、排序）放在 `site.config.yaml` → `map.nodes`：

```yaml
map:
  nodes:
    - year: 2021
      title: 湖南大学
      subtitle: 数据科学与大数据技术专业
      contentFile: experiences/university.md
    - year: 2024
      title: 字节跳动
      subtitle: 抖音电商测试开发实习
      contentFile: experiences/bytedance.md
```

示例故事（`content/experiences/baidu.md`）：

> 在百度，我开始真正接触复杂业务系统。养老业务看起来不是最酷的方向，但它让我意识到，技术真正有价值的地方，不只是高并发和复杂架构，也包括解决真实世界里具体的人和具体的问题。

模块目标：让访问者感受到站点主人不只有技术标签，还有真实业务经历与持续成长。

⸻

### 5.3 项目博物馆：项目经历展示

项目经历不做简历式列表，而是「项目博物馆」——每个项目是一个展馆。

示例项目列表（配置在 `site.config.yaml` → `projects.items`，详情在 `/content/projects/`）：

1. 智能运营平台
2. 健康管理系统
3. 老人端娱乐内容与互动服务
4. 轻量化运营交易履约系统
5. AI Agent 探索与实践

每个项目详情页统一结构（Markdown 中用二级标题分段）：

* 项目背景
* 业务问题
* 我的职责
* 技术方案
* 关键难点
* 解决过程
* 最终结果
* 我的思考

列表卡片信息来自 frontmatter，正文来自 Markdown 文件。

⸻

### 5.4 AI Agent 展区

展示站点主人对 AI Agent 的理解与实践。

内容放在 `/content/profile/agent-philosophy.md` 或独立页面 Markdown，是否在导航展示由 `site.config.yaml` → `modules.agentShowcase` 控制。

可包含：

* 什么是 AI Agent
* 为什么关注 AI Agent
* Agent 和传统后端系统的区别
* Tool Calling、MCP、Memory、Workflow、RAG
* Agent 在具体业务中的应用想象

⸻

### 5.5 Ask Me Anything：挑战数字分身

网站最核心的互动功能。访问者向个人 Agent 提问，Agent 基于站点主人维护的知识库回答。

知识库来源：`/content` 下所有 Markdown 文件，包括但不限于：

* 个人介绍（profile）
* 项目经历（projects）
* 工作经历（experiences）
* 面试问答（qa）
* 技能说明（skills）
* 职业规划、工作方式（profile / work-with-me）

Agent 回答原则（写入 `site.config.yaml` → `agent.guidelines` 并注入 Prompt 模板）：

* 不夸大经历
* 不编造不存在的项目
* 保持真实
* 回答风格接近站点主人本人
* 可适当口语化
* 对不足要诚实
* 对优势要表达清楚
* 知识库无相关内容时，明确说明「这方面资料里没有写到」

⸻

### 5.6 技术树：能力可视化

技术能力不做普通列表，而是 RPG 技能树。

**树形骨架**放在 `site.config.yaml` → `skills.tree`（分类、节点名、排序、是否可展开）。

**每个技能点的详细说明**放在 `/content/skills/*.md`，点击节点时加载对应文件或文件内锚点。

示例结构：

```
Backend
├── Golang
├── MySQL
├── Redis
└── System Design
AI
├── AI Agent
├── RAG
└── MCP
Business
├── HealthCare
└── Payment
Soft Skills
├── Problem Solving
└── Communication
```

⸻

### 5.7 如果我是你的同事

展示工作方式，不是简历式「自我评价」。

结构化小节标题放在 `site.config.yaml` → `workWithMe.sections`，各节正文放在 `/content/profile/work-with-me.md`（或用 frontmatter 拆成多文件）。

可包含：

* 我的工作方式
* 我的优势
* 我的不足
* 我希望加入什么样的团队
* 我正在补强什么能力

⸻

### 5.8 数字客厅：生活与兴趣

展示真实的人，内容轻量即可。

兴趣卡片元数据（图标、标题、排序）→ `site.config.yaml` → `life.items`
详情描述 → `/content/life/*.md`

可包含：音乐、运动、游戏、读书、宠物、博客、喜欢的句子等。

7. 技术方案设计

### 7.1 前端技术选型

推荐技术栈：

* Next.js（App Router）
* TypeScript
* Tailwind CSS
* shadcn/ui
* Framer Motion
* React Markdown

选择原因：

* Next.js 适合内容型个人网站，支持 SSG，有利于 SEO
* Tailwind + shadcn/ui 开发效率高、视觉品质好
* Framer Motion 可实现时间线、技能树等动画
* React Markdown 直接渲染 `/content` 中的内容

### 7.2 后端技术选型

**第一阶段：Next.js API Routes**（推荐 MVP）

* 前后端一体，部署简单
* 用于 Agent 对话、读取知识库、调用 LLM API

**第二阶段（可选）：Golang 后端服务**

* 独立 Agent 服务、会话管理、pgvector RAG、访问日志
* 适合后续扩展，不是 MVP 必须项

### 7.3 配置与内容加载方案

页面渲染时统一通过 `lib/site-config.ts` 读取 `site.config.yaml`，通过 `lib/markdown.ts` 读取 `/content`。

```
site.config.yaml  →  站点元数据、模块开关、导航、骨架数据
/content/**/*.md  →  详情页、故事、知识库、Agent 检索源
```

建议对 `site.config.yaml` 做 schema 校验（如 Zod），启动或构建时检查必填字段，避免配置缺失导致页面渲染异常。

### 7.4 AI Agent 技术方案

基础链路：

```
User Question
    ↓
Next.js API (/api/chat)
    ↓
读取 site.config.yaml（Agent 人设、视角规则）
    ↓
检索 /content 知识库（关键词匹配 → 后续可升级 pgvector）
    ↓
拼装 Prompt 模板（变量来自 config）
    ↓
LLM
    ↓
Answer（流式返回）
```

Agent 特点：

* 基于站点主人真实资料回答
* 不编造经历
* 可承认不知道
* 回答风格可通过 config 调整
* 可区分访问者视角（面试官 / PM / 后端 / HR / 朋友）

访问者视角配置示例（`site.config.yaml` → `agent.visitorPerspectives`）：

```yaml
agent:
  visitorPerspectives:
    interviewer:
      focus: [项目深度, 技术方案, 问题解决, 团队协作]
    productManager:
      focus: [业务问题, 用户价值, 产品链路, 协作方式]
    backendEngineer:
      focus: [架构设计, 数据模型, 性能优化, 异常处理]
```

### 7.5 site.config.yaml 设计

配置文件是整个模板的「站点控制面板」。建议提供 `site.config.example.yaml` 作为完整示例，使用者复制为 `site.config.yaml` 后修改。

推荐顶层结构：

```yaml
site:
  name: "Hello, I'm Xuhao"       # 站点标题

  highlights:
    - AI Agent

    - 健康养老业务
    - 支付与交易系统

navigation:
  modules:
    - id: map
      label: 人生地图
      path: /map
      enabled: true
    - id: projects
      label: 项目博物馆
      path: /projects
      enabled: true
    - id: agent
      label: Ask Me Anything
      path: /agent
      enabled: true
    - id: skills
      label: 技术树

      label: 如果我是你的同事

      enabled: true

      label: 数字客厅
      path: /life
      enabled: true

map:
  nodes: []        # 人生地图节点，见 5.2

projects:
  items: []        # 项目卡片元数据，slug 对应 /content/projects/{slug}.md

skills:
  tree: []         # 技能树骨架，见 5.6

  title: "如果我是你的同事"

  accent: "#6366f1"

  github: ""
  email: ""
  blog: ""
```

**配置与内容的职责划分：**

| 内容类型 | 存放位置 | 示例 |
|---------|---------|------|
| 站点标题、主人姓名 | site.config.yaml | `site.owner` |
| 首页标签、预设问题 | site.config.yaml | `hero.highlights` |
| 模块是否展示 | site.config.yaml | `navigation.modules[].enabled` |
| 人生地图节点顺序 | site.config.yaml | `map.nodes` |
| 项目卡片摘要 | frontmatter | `summary`, `tags`, `priority` |
| 项目详情八段式 | Markdown 正文 | `content/projects/*.md` |
| 技能树分类结构 | site.config.yaml | `skills.tree` |
| 技能点详细说明 | Markdown | `content/skills/*.md` |
| Agent 知识库 | Markdown | `content/` 全部 |

### 7.6 知识库设计（/content）

### 7.7 RAG 检索方案

**MVP**：读取 `/content` 下 Markdown，按关键词 + tags + type 做简单匹配。

**进阶（可选）**：PostgreSQL + pgvector，对 chunk 做 embedding 检索。

### 7.8 Agent Prompt 设计

Prompt 必须**模板化**，不可写死某个人的名字。从 `site.config.yaml` 注入变量。

示例（`lib/prompt.ts` 中的模板）：

```
你是 {{owner}} 的个人数字分身，名字叫 {{agentName}}。
你的任务是帮助访问者了解 {{owner}} 的经历、项目、技术能力、工作方式和个人特点。

回答要求：
1. 必须基于以下知识库内容回答。
2. 不允许编造不存在的经历。
3. 如果知识库中没有相关内容，要诚实说明。
4. 回答要自然、真诚、清晰。
5. 不要把自己包装成远超实际经验的人。

知识库内容：
{{retrievedChunks}}
```

站点主人的个性化回答风格，可在 `site.config.yaml` → `agent.guidelines` 中追加规则。

### 7.9 数据库设计

MVP 阶段不需要数据库。

后续如需记录会话与反馈，可增加 `conversation_sessions`、`conversation_messages`、`visitor_feedback` 等表。

8. 使用手册设计（Layer 3）

模板仓库需包含两份手册，这是降低定制门槛的关键交付物。

### 8.1 CUSTOMIZE.md（给人看）

建议包含以下章节：

1. **快速开始**：clone → 安装依赖 → 复制配置文件 → 本地预览
2. **配置文件说明**：`site.config.yaml` 每个字段的含义与示例
3. **内容文件说明**：`/content` 目录结构、frontmatter 规范、如何新增项目
4. **常见定制场景**：
   * 我只想做静态展示，不需要 Agent
   * 我想隐藏某个模块（如数字客厅）
   * 我想新增一个项目
   * 我想替换所有示例内容为我的信息
5. **Agent 配置**：如何填写知识库、如何设置预设问题、如何配置 LLM API Key
6. **部署指南**：Vercel 部署步骤与环境变量说明

### 8.2 CUSTOMIZE_FOR_AI.md（给 AI 工具看）

建议包含以下章节：

1. **项目概述**：这是一个可定制的个人数字分身模板，不是单用户私有项目
2. **可修改文件清单**：
   * `site.config.yaml`
   * `/content/**/*.md`
   * `public/images/` 中的个人图片
3. **不可随意修改的文件**：
   * `/app` 页面路由结构
   * `/components` 核心组件逻辑
   * `/lib` 工具函数（除非用户明确要求功能改造）
4. **标准定制 Prompt 模板**：

```
请根据我提供的简历 / 自我介绍，定制这个个人数字分身网站。

要求：
1. 修改 site.config.yaml 中的站点信息、hero、导航、技能树骨架、人生地图节点等结构化字段
2. 在 /content 目录中生成或更新对应的 Markdown 文件
3. 保持 frontmatter 格式正确，slug 与配置文件中的引用一致
4. 内容必须基于我提供的材料，不可编造未提及的项目或经历
5. 不要修改 /app、/components、/lib 中的代码，除非我发现页面无法正常渲染

我的简历 / 自我介绍如下：
[粘贴内容]
```

5. **质量检查清单**：定制完成后应检查的项（链接是否通、slug 是否一致、示例内容是否替换干净）

10. MVP 版本规划

### MVP 必须包含

* 完整页面路由（首页 → 数字客厅）
* `site.config.yaml` 配置体系 + schema 校验
* `/content` Markdown 内容渲染
* `site.config.example.yaml` 示例配置（周旭豪示例）
* `/content` 示例内容（可基于本文档示例）
* `CUSTOMIZE.md` + `CUSTOMIZE_FOR_AI.md`
* 个人 Agent 问答（/api/chat + ChatBox）
* 技术树、项目博物馆、人生地图基础版

### MVP 不做

* 简历上传与大模型自动解析
* 在线内容编辑器 / 后台管理系统
* 登录系统
* 复杂向量库（pgvector）
* 复杂动画
* 访问数据分析
* 多语言

### MVP 技术实现

* Next.js + TypeScript + Tailwind CSS + shadcn/ui
* `site.config.yaml` + `/content` Markdown
* LLM API + 简单关键词 RAG
* Vercel 部署

### 使用者上手流程（MVP 目标体验）

```
1. git clone 项目
2. cp site.config.example.yaml site.config.yaml
3. 按 CUSTOMIZE.md 替换配置与内容
   （或把 CUSTOMIZE_FOR_AI.md + 简历交给 Cursor 辅助生成）
4. pnpm dev 本地预览
5. vercel deploy 上线
```

14. 项目最终目标

对**站点主人（模板使用者）**：

* fork 模板后，用配置 + Markdown 低成本完成个人站定制
* 不需要从零设计信息架构，也不需要写大量页面代码
* 可选择手动定制，或用 AI 工具辅助改写内容文件

对**站点访问者**：

* 不是「看完一份简历」，而是「认识一个具体的人」
* 可以浏览，也可以对话

这个模板应该能同时帮助任何一个使用者回答三个问题：

1. 我是谁？
2. 我做过什么？
3. 为什么值得和我聊一聊？

如果做得好，它既可以作为求职作品，也可以长期作为个人品牌入口，还可以作为开源模板帮助他人快速搭建自己的数字分身。

它不是一份简历。

它是每个人的个人操作系统——而这套代码，就是安装向导。
